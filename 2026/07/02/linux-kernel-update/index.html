Linux服务器 系统内核升级完整指南

在长期运行的 Linux 服务器上，经常会遇到应用程序要求更高版本内核的场景。尤其是当宿主机仍使用 Ubuntu 18.04 一类较老的发行版，而业务侧又依赖 io_uring、新版 eBPF 或较新的驱动特性时，往往必须在不整体升级系统的前提下更换内核。整个过程涉及内核包本身、glibc、编译工具链以及 NVIDIA DKMS 模块之间的多层依赖，任一环节处理不当都可能导致重启后系统无法正常进入图形界面或 GPU 驱动失效。

本文以 Ubuntu 18.04 + NVIDIA 驱动 + 目标内核 5.5.5 （升级前：4.15.0） 的实际升级过程为例，梳理内核升级涉及的核心概念、约束条件、完整执行流程以及常见问题的排查方法，供有类似需求的运维与开发人员参考。

1 Linux 内核与相关组件

1.1 内核版本号的含义

Linux 内核（Linux Kernel）是操作系统的核心，负责管理硬件资源（CPU、内存、磁盘、网卡、GPU 等）、调度进程并提供系统调用接口。执行 uname -r 得到的字符串即为当前运行的内核版本，例如 4.15.0-55-generic：

版本号的语义大致如下：

  • 主版本号变化（4.x → 5.x）通常伴随架构性改动；
  • 次版本号变化（5.4 → 5.5）意味着新增功能、驱动或系统调用；
  • 偶数次版本号（如 5.4、5.10、5.15）通常对应 LTS（长期支持）内核；
  • 奇数次版本号（如 5.5、5.7、5.9）通常为开发主线，支持周期较短。

1.2 内核与驱动的协作关系

内核版本与驱动版本是两个层次上的概念，在实际运维中经常被混淆。以 NVIDIA 为例：

概念 版本号示例 作用
Linux 内核 4.15.0-55-generic 操作系统核心，管理所有硬件
NVIDIA 用户态驱动 550.54.14 提供 CUDA、OpenGL 等用户态库
NVIDIA 内核模块（nvidia.ko） 同上 内核态驱动，负责内核与 GPU 之间的通信

NVIDIA 内核模块依附于当前运行的内核，每次更换内核版本都需要重新编译一次。这一过程通常由 DKMS（Dynamic Kernel Module Support） 自动完成。若 DKMS 在新内核上编译失败，重启后 nvidia-smi 将无法与驱动通信。

2.1 程序对新内核特性的硬性依赖

此类硬性依赖通常源自新内核引入的特性，包括：

  • io_uring：高性能异步 I/O 接口，5.1 引入，5.5 之后趋于稳定；
  • BPF / eBPF：5.x 之后功能大幅增强；
  • cgroup v2：5.x 之后默认启用，部分容器与调度框架依赖；
  • 新增的 syscall，例如 clone3、openat2；
  • 新硬件的驱动支持，例如较新的 GPU 与网卡。

2.2 安全补丁与生命周期

较早的内核不再接收安全更新。Ubuntu 18.04 自带的 4.15 内核在 2023 年 5 月已结束标准支持（EOL），此后发现的 CVE 不会被官方修复。对于暴露在公网或承担生产业务的机器，长期停留在 EOL 内核会引入较大的安全风险。

2.3 新硬件的驱动支持

新的 CPU、GPU 与网卡对应的驱动往往仅存在于较新的内核中。部分 2022 年之后发布的网卡芯片，在 4.15 内核下无法被识别。

2.4 性能改进

新版内核通常带来调度器、内存管理以及 ext4 / xfs / btrfs 等文件系统层面的性能优化。对于高并发或高 I/O 场景，升级内核可以带来约 10% 以上的吞吐提升。

3 升级方案的选择

升级内核主要有三条路径，各自适用于不同场景：

方案 速度 风险 可回滚 适用场景
HWE（Hardware Enablement） 快 低 易 同系列内小版本提升（如 4.15 → 5.4）
主线 PPA 内核 中 中 易 需要特定版本，系统不能整体升级
升级整个 Ubuntu 发行版 慢 高 难 系统已 EOL，希望一次性解决

3.1 老系统上升级内核的核心约束

在较老的系统上升级内核，会在三个层面同时受到约束，每一项都可能导致升级失败。理解这些约束是判断目标内核是否可行的基础。

3.1.1 glibc 版本约束

Ubuntu 主线内核 PPA 上的 linux-headers deb 包会随编译环境一同打包。编译时间越晚，包对 glibc 的依赖越高。以 Ubuntu 18.04（glibc 2.27）为例：

内核版本 headers 包的 glibc 依赖 是否可在 18.04 上安装
5.5.5（2020-02 编译） libc6 (>= 2.14) 可
5.15.0（2021-10 编译） libc6 (>= 2.34) 否

glibc 是系统最底层的 C 库，几乎所有用户态程序都依赖它，强行升级会导致包括 ls 在内的基础命令都无法运行。因此 glibc 的版本基本等同于 Ubuntu 发行版的版本：

Ubuntu 版本 glibc 版本
18.04 2.27
20.04 2.31
22.04 2.35

对应的结论是：在 Ubuntu 18.04 上，主线 PPA 中 5.15 以后的 headers 无法安装，只能选择 2020 年初编译的 5.5.x 一类的“窗口期”版本。

3.1.3 HWE 的版本天花板

HWE（Hardware Enablement） 是 Ubuntu 官方提供的新内核通道，可以通过 apt 直接安装，稳定性优于主线 PPA。但 HWE 存在版本上限：

Ubuntu 版本 默认内核 HWE 版本上限
18.04 4.15 5.4（HWE edge）
20.04 5.4 5.15（HWE edge）
22.04 5.15 6.8（HWE edge）

若目标版本高于 HWE 上限，只能选择主线 PPA 或整体升级发行版。

5.1 升级内核后 nvidia-smi 报 “couldn’t communicate with the NVIDIA driver”

该问题通常源于新内核的 nvidia 模块未能成功编译，或编译成功后未加载。排查步骤如下：

如需临时回退，可在重启时于 GRUB 菜单中选择旧内核。

5.2 DKMS 报 apport 相关错误

若日志中出现类似 ERROR (dkms apport): binary package for nvidia: 550.54.14 not found 的错误，该信息本身并非编译失败的真实原因，而是 apport 尝试生成错误报告时未找到对应 deb 包所致。真实错误需要查看 make.log。

5.3 主线 PPA 下载 404

原因通常是 PPA 路径结构发生变化。旧路径 /~kernel-ppa/mainline/vX.Y.Z/amd64/file.deb 虽然会 301 重定向，但新路径下不存在 /amd64/ 子目录。

正确的路径为 https://kernel.ubuntu.com/mainline/vX.Y.Z/file.deb，架构由文件名末尾的 _amd64.deb 区分。

5.4 glibc 依赖不满足

若 dpkg 报出类似以下错误：

说明目标内核的 headers 包要求的 glibc 版本高于当前系统。由于 glibc 无法独立升级，可选的处理方式为：

  1. 选择编译时间更早的内核包（例如 5.5.x 而非 5.15.x）；
  2. 整体升级 Ubuntu 发行版（18.04 → 20.04 → 22.04）。

5.5 DKMS 编译报 expected '(' before '__inline'

Linux 5.4 至 5.6 内核中的 asm_inline 宏语法仅在 GCC 9 及以上版本中被支持。若系统 GCC 仍为 7.x，需要按 4.2 节的方式升级 GCC，并确认 DKMS 实际调用的是新版 GCC（应通过 /usr/bin/gcc --version 确认，而非 which gcc）。

5.7 删除内核时出现 “unbootable” 警告

该警告由 apt 的系统保护机制触发，原因是尝试删除当前正在运行的内核。正确的处理顺序是先切换到其他内核启动，再删除不再需要的内核，即“先切换、后删除”。

5.8 HWE 安装的内核版本不够高

HWE 存在版本上限。例如 Ubuntu 18.04 的 HWE edge 最高只能到 5.4.x，无法满足 5.5+ 的需求。此时需要通过主线 PPA 安装指定版本，或整体升级 Ubuntu。

5.9 升级内核与升级发行版的选择

两种方式各有其适用场景：

情况 推荐方式
程序硬性依赖新内核，系统整体可用 通过主线 PPA 升级内核
系统已 EOL，希望长期稳定 升级 Ubuntu（18.04 → 20.04 → 22.04）
短期需要顶住，长期希望彻底解决 先用主线 PPA 满足需求，再规划系统升级

Ubuntu 跨大版本升级必须逐级进行，不支持跨版本跳升（需 18.04 → 20.04 → 22.04）。

5.10 主线内核与 HWE 内核的差异

特性 HWE 内核 主线 PPA 内核
来源 Ubuntu 官方维护 Ubuntu 内核团队构建，非官方支持
安全更新 有（随 Ubuntu LTS 支持周期） 无（单次快照，不再更新）
版本选择 受限于 Ubuntu 版本 可选任意版本
适用场景 生产环境 临时需求或测试

主线内核可以理解为“快照”，安装之后不会再收到安全补丁，适合临时满足需求，但不建议长期用于生产。

6 总结

在较老的发行版上升级内核，本质上是在多个约束条件之间寻找平衡：

在本文的案例中，5.5.5 恰好是 Ubuntu 18.04 上能够同时满足这三项约束的最低 5.5+ 内核版本：

  • glibc 2.27 ≥ 2.14（headers 可安装）；
  • GCC 9.5.0（asm_inline 可编译）；
  • 主线 PPA 不受 HWE 版本上限限制。

需要指出的是，5.5.5 属于非 LTS 版本，不再接收安全更新。长期方案仍应是整体升级到 Ubuntu 22.04（默认内核 5.15 LTS），从根本上避免 glibc、GCC、内核版本互相牵制的问题。
